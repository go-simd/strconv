# strconv

[![CI](https://github.com/go-simd/strconv/actions/workflows/ci.yml/badge.svg)](https://github.com/go-simd/strconv/actions/workflows/ci.yml)
![coverage](https://img.shields.io/badge/coverage-100%25-brightgreen)

A drop-in fast path for **base-10 integer and float parsing** whose results â€”
both the value **and** the error â€” are byte-identical to the standard library's
[`strconv`](https://pkg.go.dev/strconv). The hot case (a clean run of ASCII
decimal digits, base 10) is crunched by a SIMD kernel generated by
[go-asmgen](https://github.com/go-asmgen/asmgen) (amd64 SSE, ppc64le VSX, s390x
vector facility; other architectures use a tight scalar loop). Everything
ambiguous is **delegated verbatim to
`strconv`**, so the returned value and the exact `*strconv.NumError` match. Pure
Go, `CGO_ENABLED=0`, stable Go, no `GOEXPERIMENT`.

```go
v, err := strconv.ParseUint(s, 10, 64) // same value + same error as strconv.ParseUint
n, err := strconv.Atoi(s)              // same value + same error as strconv.Atoi
i, err := strconv.ParseInt(s, 10, 64)  // same value + same error as strconv.ParseInt
f, err := strconv.ParseFloat(s, 64)    // same float bits + same error as strconv.ParseFloat
```

API drop-ins for `strconv`: `ParseUint(s string, base, bitSize int) (uint64, error)`,
`ParseInt(s string, base, bitSize int) (int64, error)`, `Atoi(s string) (int, error)`,
`ParseFloat(s string, bitSize int) (float64, error)` (`bitSize` 32 and 64).

## Correctness first

Correctness is paramount: a correct-but-modest result beats a fast-but-wrong
one. The SIMD fast path is taken **only** for the unambiguous case and only ever
returns a `nil` error:

- base 10 (base 0 with no prefix is left to `strconv`), `bitSize` 0 or 64;
- an optional leading `+`/`-` (handled scalarly, signed parsers only), then
- a pure run of decimal digits whose value cannot overflow (length-bounded), and
- long enough to be worth the SIMD call (see *Performance*).

**Anything else** â€” other bases, base prefixes (`0x`/`0o`/`0b`), `_` digit
separators, empty input, a non-digit byte, overflow, over-long input, unusual
`bitSize` â€” falls back to the real `strconv`. So the value and the full
`*strconv.NumError` (`Func`, quoted `Num`, `ErrSyntax`/`ErrRange`/â€¦) are exactly
what `strconv` would return.

This equivalence is proven by `FuzzParseUint`, `FuzzAtoi` and `FuzzParseInt`,
which compare the returned **value AND the full error string** against `strconv`
on arbitrary input, plus an exhaustive table (zero; every length 1..21; leading
zeros; signs; MaxUint64 / MaxInt64 / MinInt64 boundaries and one past them;
overflow; empty; a non-digit at every position; `_` separators; bases â‰  10).
The suite passes natively on arm64 and amd64, under amd64 SSE (Rosetta + native
CI), and â€” for the VSX and z/Architecture vector kernels â€” under qemu
(`ppc64le`/`power9` and `s390x`) in the emulated CI job, where all four fuzzers
run against the native cross-toolchain too. 100 % statement coverage on every
architecture.

### `ParseFloat`: the Eisel-Lemire fast path + a hard delegate discipline

`ParseFloat` extends the same discipline to decimal floats. For a clean
input â€” an optional sign, decimal digits with at most one `.`, and an optional
`e`/`E` exponent â€” it folds the significant mantissa (the `parse16SSE` kernel
when a clean â‰Ą 16-digit run is present) into a 64-bit integer and a base-10
exponent **exactly the way the standard library's `readFloat` does**, then
converts via the very same `atofNNexact` / **Eisel-Lemire** routines `strconv`
uses (vendored verbatim from `internal/strconv`, BSD-licensed). Those routines
are *self-limiting*: when the cheap 128-bit approximation cannot **prove** the
correctly-rounded result, they decline â€” and we delegate. We never round
ourselves, so the returned float bits are bit-identical to `strconv`.

**The fast path returns a value only when it can prove it equals `strconv`'s.**
It **delegates** (verbatim, to `strconv.ParseFloat`) for *every* uncertain case:
hex floats (`0xâ€¦pâ€¦`), `inf`/`Inf`/`Infinity`/`nan` (any case), exponent-only /
leading-dot / trailing-dot oddities, `_` digit separators, empty or malformed
input, `bitSize` â‰  32/64, mantissas truncated past the 19-digit window whose
upper bound it cannot confirm, absurd exponents, and **any input near a rounding
boundary**. Delegation is *correct*, not a failure.

This is proven by **`FuzzParseFloat`**, which compares the IEEE-754 **bits**
(`math.Float64bits`/`Float32bits`, so sign-of-zero, NaN and exact rounding are
caught) **and the full error string** against `strconv.ParseFloat` for **both
`bitSize` 32 and 64** on arbitrary input â€” > 100 M executions clean natively
(arm64) and on a real-AVX2 x86_64 VM, plus a broad table (integers-as-floats,
fractions, scientific notation, Â±0.0, subnormals, `MaxFloat64` /
`SmallestNonzeroFloat64` boundaries, overflow â†’ Â±Inf + `ErrRange`, underflow â†’ 0,
every malformed form, hex, inf/nan, `_`).

The 16-digit decimal fold is implemented for six 64-bit targets. The amd64,
ppc64le and s390x kernels are SIMD; arm64, loong64 and riscv64 use the scalar
fold (NEON/LSX/RVV ports planned). ppc64le and s390x are **qemu-validated
(native perf pending)**; amd64 and arm64 run on native CI.

| op | amd64 | ppc64le | s390x | arm64 / loong64 / riscv64 |
|---|---|---|---|---|
| **ParseUint / ParseInt / Atoi (base 10)** | **SSE2/SSSE3** 16-digit kernel | **VSX** 16-digit kernel | **vector facility** 16-digit kernel | scalar â€” NEON/LSX/RVV planned |
| **ParseFloat (decimal, bitSize 32/64)** | **SSE2/SSSE3** mantissa fold + Eisel-Lemire | **VSX** mantissa fold + Eisel-Lemire | **vector** mantissa fold + Eisel-Lemire | scalar fold + Eisel-Lemire |

The VSX and z/Architecture kernels mirror the amd64 staged multiply-accumulate.
Neither ISA exposes (in Go's assembler) a single `pmaddwd`-style fused op, so each
fold stage is a pair of even/odd **widening integer vector multiplies** plus an
add â€” `VMULEUB`/`VMULOUB` â†’ `VMULEUH`/`VMULOUH` â†’ `VMULEUW`/`VMULOUW` on POWER,
and `VMLEB`/`VMLOB` â†’ `VMLEH`/`VMLOH` â†’ `VMLEF`/`VMLOF` on z. The digit string is
loaded with `LXVB16X` (ppc64le) / `VL` (s390x); on both, element 0 is the most
significant digit, and **s390x is big-endian** â€” its power-of-ten weights are
laid out most-significant-first to match, pinned by a position-dependent test.

## How it works

The Lemire "parse a number at a gigabyte per second" SIMD decimal core, folding a
whole 16-digit group in **one** assembly call (on amd64 SSE):

- **Load** the 16 chars into an XMM register (`MOVOU`).
- **Validate** â€” a char is a decimal digit iff `('0'-1) < c < ('9'+1)`, tested
  with two `PCMPGTB` (signed; digit chars are `< 0x80` so the sign bit is clear).
  All sixteen lanes must pass: `PMOVMSKB` of the isDigit mask must be `0xFFFF`,
  else the group is rejected (the Go wrapper then re-walks it scalarly so a
  non-digit ends the run exactly as `strconv`).
- **Fuse** digits â†’ value (`PSUBB '0'` first, giving 16 digit bytes 0..9):
  - `PMADDUBSW [10,1,â€¦]` : 16 digit bytes â†’ 8 words, each a 2-digit value
    `10Â·hi + lo`.
  - `PMADDWL [100,1,â€¦]` : 8 words â†’ 4 dwords, each a 4-digit value `100Â·hi + lo`
    (`< 10000`). `PMADDWL` is Go's mnemonic for SSE2 **`pmaddwd`** â€” it does the
    staged wordâ†’dword fold in a single instruction (see the note below).
  - The four 4-digit dwords `c0..c3` are combined in GP registers as
    `hi = c0Â·10000 + c1`, `lo = c2Â·10000 + c3`, `val = hiÂ·1e8 + lo` â€” three
    multiplies, keeping the assembly trivially correct.

The Go wrapper makes one `parse16SSE` call and finishes the 0..3-digit tail
(inputs are 16..19 digits) with a scalar loop. The kernel takes a `*byte` (via
`unsafe.StringData`), so the hot path performs **no `[]byte(s)` allocation**.

The single-call 16-digit kernel replaced an earlier two-call 8-digit loop: the
dominant cost for these short fixed-length inputs is the non-inlinable assembly
**CALL** (frame setup, register spills around the call, ABI0 wrapper), not the
digit crunch â€” so collapsing two calls into one roughly **doubled** the win
(see *Performance*). `llvm-mca` (Haswell model) also confirms the compute halved:
the 16-digit fold is ~7 cycles in one block vs ~10 cycles across two 8-digit
blocks.

### PMADDWL (a previously-thought-missing mnemonic)

The canonical Lemire kernel fuses the 2-digit words into 4-digit dwords with one
**`pmaddwd`** (SSE2, `[100,1,100,1]`). An earlier version of this package used a
`PMULLW`+`PHADDW` workaround, believing Go's assembler lacked the SSE2 form. It
does **not**: Go spells SSE2 `pmaddwd` as **`PMADDWL`** (and the AVX2 form as
`VPMADDWD`), both present in `cmd/internal/obj/x86`. The kernel now uses
`PMADDWL` directly â€” fewer ops on the critical path.

## Performance

Honest numbers, **native amd64**, `go1.26.4`, median of 8, `benchtime=0.5s`, on a
QEMU/KVM Haswell guest (the dev box is arm64; the amd64 kernel only runs under
Rosetta locally, which is unrepresentative, so this uses a native x86_64 VM â€” CI
reproduces it on GitHub `ubuntu-latest`). `GOAMD64=v3`. Numbers below are the
inputs where the SIMD path is actually taken:

| call | SIMD | `strconv` | speedup |
|---|---:|---:|---:|
| `ParseUint` 16 digits | **75 ns** | 171 ns | **2.29Ă—** |
| `ParseUint` 17 digits | **80 ns** | 178 ns | **2.22Ă—** |
| `ParseUint` 18 digits | **82 ns** | 187 ns | **2.29Ă—** |
| `ParseUint` 19 digits | **87 ns** | 190 ns | **2.19Ă—** |
| `Atoi` 19 digits | **92 ns** | 244 ns | **2.66Ă—** |
| `ParseUint`/`Atoi` â‰¤ 8 digits | delegated â†’ `strconv` | â€” | 1.00Ă— (no regression) |
| `ParseFloat` `3.14159` | **174 ns** | 237 ns | **1.36Ă—** |
| `ParseFloat` `1e308` | **165 ns** | 222 ns | **1.34Ă—** |
| `ParseFloat` `1234567890.12345` (15 sig) | **318 ns** | 358 ns | **1.13Ă—** |
| `ParseFloat` `1.7976931348623157` (17 sig) | **361 ns** | 393 ns | **1.09Ă—** |

**Ratio re-confirmation (as of 2026-06-14):** re-benched `-count=6` medians on a
QEMU x86_64 lima VM (absolutes are TCG-low and not comparable to the table above,
but the SIMD/stdlib *ratio* is valid). `ParseUint` **16-digit: ~73 ns SIMD vs
~179 ns stdlib = ~2.45Ă—**; **18-digit: ~80.5 vs ~195 = ~2.42Ă—**; **8-digit:
delegated, stdlib slightly ahead (~0.86Ă—) â€” the documented no-win case below the
16-byte window.** On **native arm64** (this host) there is *no* SIMD kernel
(scalar fold), yet 16-digit still measures **~2.0Ă—** stdlib (~8.0 vs ~16.5 ns) â€”
that win is our tighter scalar fast path vs `strconv`'s per-call bookkeeping, not
vectorisation. Verdict unchanged: â‰Ą16-digit SIMD beats stdlib ~2.4Ă— on amd64,
~2.0Ă— on arm64; <16 digits is delegated (no regression).

Honest findings â€” this is where SIMD does and does **not** pay:

- **The win is large on long inputs (16â€“19 digits): ~2.2Ă— for `ParseUint`,
  ~2.7Ă— for `Atoi` at 19 digits.** Collapsing the old two-call 8-digit loop into
  a single 16-digit `parse16SSE` call removed the dominant cost â€” the
  non-inlinable assembly **CALL** and its register-spill / frame-reload overhead
  â€” roughly doubling the previous 1.1â€“1.4Ă— margins.
- **Short inputs cannot win, and we don't try.** Go's `strconv` scalar parsers
  are already very tight, and for < 16 digits there is no full 16-byte group to
  vectorise; a single non-inlinable assembly CALL costs more than stdlib's
  inlined scalar loop. So we **delegate every < 16-digit input to `strconv`**,
  which is both correct and faster â€” the package never regresses there.
- **`strconv.Atoi` is special.** It has an *inlined* digit-at-a-time fast path
  for inputs **shorter than 19 digits** (64-bit) which SIMD cannot beat. Only at
  exactly 19 digits does `Atoi` drop to its slower `ParseInt` path â€” and there
  SIMD wins **2.66Ă—**. So `Atoi` takes the SIMD path **only at 19 digits**
  (`minAtoiDigits`); `ParseUint`/`ParseInt`, which have no such inlined path,
  take it from 16 digits (`minFastDigits`).
- **20-digit and overflow inputs** delegate to `strconv` for its exact
  `MaxUint64`/`ErrRange` behaviour (no fast path, no regression).
- **`ParseFloat` wins ~1.1â€“1.4Ă—** on clean decimals by skipping `strconv`'s
  `special()` / `readFloat` dispatch and going straight to the shared
  Eisel-Lemire conversion; the SIMD 16-digit mantissa fold adds margin when a
  clean â‰Ą 16-digit integer run precedes any `.`. The win is modest, not 2Ă—,
  because the float math (the 64Ă—64â†’128 multiply + rounding proof) dominates the
  digit fold â€” reported honestly. **Rounding-boundary, truncated, hex, inf/nan
  and malformed inputs delegate to `strconv`** (correct by construction, never a
  regression).

The takeaway, reported rigorously: this is a **correct** drop-in (fuzz-proven
value + error equality vs `strconv`) with a **substantial, honest** SIMD win
(~2.2â€“2.7Ă—) on long base-10 inputs, and **parity (never a regression)** on short
inputs via fallback. Short-string scalar parsing does not benefit from SIMD â€”
that is the honest result, and the package delegates those cases so it never
loses.

### ppc64le / s390x â€” llvm-mca cycle-model estimate

**Static analysis, NOT a hardware measurement; native perf pending real silicon.**
No native POWER/Z runner is available and QEMU is not cycle-accurate, so the
defensible perf signal is a cycle-model estimate. Unlike the streaming kernels,
`parse16` is a **straight-line, single-call helper** (it folds one 16-digit group
per call, no inner loop), so the meaningful figure is the **single-invocation
critical-path latency**, not a steady-state throughput ceiling. The committed
bodies were extracted from `parse_ppc64le.s` / `parse_s390x.s` and run through
`llvm-mca -iterations=1` (LLVM 22; production PowerPC + SystemZ backends):

```
llvm-mca -mtriple=powerpc64le-unknown-linux-gnu -mcpu=pwr9 -iterations=1 <parse16.s>
llvm-mca -mtriple=s390x-unknown-linux-gnu -mcpu=z14  -iterations=1 <parse16.s>
```

The Ă—scalar baseline is the classic `n = n*10 + (c-'0')` loop unrolled over the
same 16 digits (a 16-deep dependent multiply-add chain):

| arch (cpu) | SIMD `parse16` latency | scalar 16-digit latency | est. Ă—scalar |
|---|---:|---:|---:|
| ppc64le (pwr9) | ~47 cyc | ~116 cyc | **~2.5Ă—** |
| s390x (z14)    | ~39 cyc | ~84 cyc  | **~2.2Ă—** |

The win comes from collapsing the **16-deep serial `n*10+d` dependency chain**
into a **logarithmic tree** of pairwise widening multiplies (`VMULEUB`/`VMULOUB`
â†’ `VMULEUH`/`VMULOUH` â†’ `VMULEUW`/`VMULOUW` on POWER; `VMLEB`/`VMLOB` â†’
`VMLEH`/`VMLOH` â†’ `VMLEF`/`VMLOF` on Z) plus a single `VCHLB`/`VCMPGTUB`
range-check instead of 16 per-digit compares â€” depth ~logâ‚‚16 vs 16. The two
arches land close to each other and to the measured amd64 ~2.4Ă— (same algorithm).
Caveats: latency-only model (no cache/front-end/branch); the per-call constant
loads (`subD`/`nine`/weights) are excluded as they amortise across calls, and the
final `MFVSRD`/`VLGVG` GPR-extract + `MULLD` combine are included; the scalar
baseline is an idealised unrolled chain (real Go scalar with bounds/error checks
would be slower â†’ Ă—scalar is a conservative lower bound). All instructions are
modelled by llvm-mca. Ballpark only â€” to be replaced by native timings on real
POWER9 / z14.

## Regenerating the assembly

`parse_amd64.s` is committed. To regenerate (go-asmgen is a build-time tool, not
a runtime dependency):

```sh
go get github.com/go-asmgen/asmgen@v0.4.0
go run parse_gen.go
go mod edit -droprequire github.com/go-asmgen/asmgen
go mod tidy
```

## Coverage

The CI gate enforces **100% coverage of the Go code** on each arch job (native
amd64 + native arm64; the `!amd64` generic fallback compiles and is measured on
arm64). The tests exercise both the SIMD fast path AND every fallback-to-stdlib
branch (other bases, prefixes, `_` separators, empty input, non-digit bytes,
overflow at the signed/unsigned boundaries, over-long input, unusual bitSizes).
Coverage is of the Go statements only: the generated `.s` parse kernel is not
measured by `go test -cover` â€” it is validated by differential tests against
the scalar reference and the standard library, plus fuzzing.
